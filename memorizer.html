CASE 02 / 2D PUZZLE PLATFORMER

MEMORIZER Record / Replay Puzzle System

类型: 2D 平台解谜游戏

核心: Record / Replay 时间能力

重点: 核心玩法 / 关卡递进 / UI 反馈 / 技术落地 / 测试迭代

REPLAY

PROJECT GOAL

目标不是做一款“时间题材”的平台游戏，而是让玩家真正操作“过去的自己”来解谜。

在《Memorizer》里，整个体验建立在 `Record` 和 `Replay` 上。玩家需要先记录自己的动作，再在合适的时机把这段过去重新播放出来，让“已经发生过的行为”变成当前空间里的解法。

这个项目最吸引人的地方，在于它不是简单增加一个技能按钮，而是要求关卡、引导、操作反馈到叙事推进都围绕同一套系统来思考。只要其中一个环节没有对齐，玩家就会觉得这只是个概念，而不是一个真正成立的游戏机制。

“重点在于，玩家能不能在第一次成功利用 Replay 解开谜题时，清楚意识到自己正在和‘过去的自己’协作。”

INSPIRATION

灵感来源

最初希望围绕“时间”做一款 2D 平台解谜游戏，但不想重复常见的倒流时间或双时间线切换。更感兴趣的是: 如果玩家能够把自己的动作存下来，再在之后重新调用，这会不会形成一种更主动、更具策略性的解谜体验？

因此项目参考了《Braid》对时间机制的思考方式，《Titanfall 2》对玩法教学的节奏控制，以及《Super Mario Bros. Wonder》那种轻盈、清晰、充满可读性的关卡表达。

BACKGROUND

故事与角色

主角被设定为森林中的红蘑菇女孩伊莉斯。她意外得到一只神奇怀表，从而获得了记录与重放动作的能力。这个设定的作用不是单纯补一层剧情包装，而是帮助时间能力自然地嵌进世界观里。

叙事在这里的任务，是给玩家一个持续前进的理由: 体验不是孤立地解一串谜题，而是在逐步进入一个拥有规则、敌人和秘密的完整世界。

GAMEPLAY RULES

玩法规则被尽量设计得明确: 先记录，再重放，然后让两段时间在空间里互相配合。

RULE 01

Record 阶段

在这个阶段里，玩家会把自己的移动、跳跃和关键操作记录下来。这个过程需要尽可能直接，所以 Record 本身不应该让玩家产生额外负担，它更像是在告诉玩家:“你现在做的每个动作，之后都可以变成线索或工具。”

RULE 02

Replay 阶段

当玩家触发 Replay 时，之前记录过的动作会被重新播放。这个时候真正有趣的地方才开始出现: 玩家当前控制的角色，可以和被重放出来的过去动作形成配合，从而到达原本无法抵达的平台、开关或路线。

RULE 03

规则成立的关键

这套玩法不是“加了两个按钮”就结束了。它必须满足三个条件: 玩家能理解 Record 和 Replay 的区别，能预判 Replay 的结果，并愿意围绕这个结果重新规划路线。只有这三件事同时成立，这个系统才真正变成“玩法规则”，而不是单次噱头。

A1

为什么要先 Record 再 Replay

这里刻意避免让玩家直接获得一个万能回放技能，而是要求他们先做出选择，再承担这个选择的结果。这样一来，解谜的重点就从“反应快不快”转向“规划有没有想清楚”。

B2

为什么强调动作可读性

因为如果玩家看不懂 Replay 到底重放了什么动作，就无法建立稳定心智模型。因此 UI、按钮图标、动作轨迹和时机反馈都服务于这一点。

C3

为什么要把输入说明放进关卡里

为了避免玩家在页面外背说明书，控制教学被拆进了关卡推进里。也就是说，操作说明本身就是关卡设计的一部分。

LEVEL DESIGN

四个关卡层层递进，把“学会操作”逐步推进成“真正理解系统并运用它”。

LEVEL 1

把基础移动和核心机制绑在一起

第一关先教玩家移动、跳跃，再引入第一次必须使用 Record / Replay 才能通过的障碍。这里故意让奖励、路径和隐藏平台都围绕这个初次体验展开，目的是让玩家尽快意识到: 这套能力不是附加项，而是整款游戏的基础语言。

LEVEL 2

加入梯子和误导性变化

第二关引入梯子后，重点不只是难度提升，而是开始加入“看似相似、实则有细微差异”的谜题。这样做的目的是让玩家放弃机械记忆，转而真正思考 Record 阶段里到底需要做什么动作，Replay 才会在后面成立。

LEVEL 3

移动平台与垂直平台

到第三关，平台本身开始成为 Record / Replay 系统的一部分。玩家不再只是记住角色的动作，而是要考虑环境何时被操控、平台在何时处于什么位置。这样谜题就从“角色解谜”升级为“角色与环境共同解谜”。

LEVEL 4 / BOSS

把前面学过的一切集中兑现

最后 Boss 关不是单纯提高数值难度，而是把前面学过的移动、平台控制、时序判断和 Replay 协作全部集中起来。这一关的意义在于证明整套系统可以支撑完整体验，而不仅是局部谜题。

DESIGN DECISION

为什么这样安排教学和节奏

这个项目最重要的设计任务之一，是控制玩家第一次接触核心机制时的理解成本。Record / Replay 本身并不难懂，但玩家很容易在“会按按钮”和“真的明白这套系统怎么工作”之间卡住。

所以教学被拆成许多很小的、能立刻验证的瞬间。每次引入新元素时，都会先给一个低风险的成功案例，让玩家迅速获得“原来如此”的反馈，再逐渐加入变化和误导。

UI & FLOW

界面与流程也必须服务机制

UI 中加入了动态背景、呼吸式标题变化、机制触发时的画面反馈和清晰的输入提示，这些都不是装饰，而是在帮玩家不断确认当前自己处于哪个阶段、刚刚发生了什么、接下来应该尝试什么。

关卡流同样重要。每一关都要让玩家明确知道“这一关新增了什么”、 “要把什么旧知识和新知识连接起来”，这样系统才能稳定建立，而不会被误以为只是随机试错。

PROBLEM-SOLVING

技术落地过程中，最难解决的是让 Replay 真的可信

一开始，曾尝试直接记录角色的相对位置，再在 Replay 阶段把这些位置重新播放出来。但很快就遇到了严重问题: 角色会穿模、会在空中不断下落，甚至会出现 Record 和 Replay 结果不对称的情况。

之后调整思路，开始记录更多和物理状态相关的内容，例如 `velocity`、`ground check` 和 `gravityScale` 等参数。这样虽然增加了系统复杂度，但 Replay 的稳定性才真正建立起来。

最后，还处理了帧率对重放精度的影响，把很多最难察觉却最致命的问题压到了可控范围内。这部分工作直接决定了玩法是否成立，因为只要 Replay 不可靠，前面所有关卡设计都会失去说服力。

TEST & ITERATION

测试结果说明: 好的机制还需要好的引导

经过多轮测试，逐渐发现真正拖慢玩家的并不总是难度本身，而是他们第一次接触 Record / Replay 时是否能快速理解“这套规则到底想让玩家做什么”。因此后续又回过头优化了很多教程节点和提示方式。

未来如果继续扩展，希望把这套系统进一步拟人化，让 Record / Replay 不只作为 UI 元素出现，而是变成故事里真正有身份的存在。同时，也希望把这套玩法推进到更复杂的场景交互上，例如钥匙、门、机关或者可被重复利用的环境对象。
